# Design: Extract Benchmark Suite to Independent Repository

**Date:** 2026-03-26
**Status:** Approved
**Author:** zazabap

## Problem

The benchmark suite in `examples/benchmark/` has grown to ~1400 lines of code with 12 GB of data. It is a self-contained Julia project with its own `Project.toml` and has no coupling to the core library code. Keeping it in the main repo bloats clones and conflates two different change cadences.

## Decision Summary

- **New repo:** `zazabap/ParametricDFT-Benchmarks.jl`
- **History:** Preserved via `git filter-repo --subdirectory-filter`
- **Source branch:** `feature/multi-dataset-benchmark`
- **Parent repo integration:** Git submodule at `examples/benchmark`
- **Dependency:** ParametricDFT.jl via git URL (pinned to `main`)
- **Results:** Tracked in the new repo's git history
- **Data:** Remains gitignored (downloaded at runtime)
- **Minimum Julia version:** 1.11+ (required for `[sources]` in `Project.toml`)

## Prerequisites

Before starting, ensure:
- `git-filter-repo` is installed (`pip install git-filter-repo`)
- `gh` CLI is authenticated (`gh auth status`)
- User has push access to create `zazabap/ParametricDFT-Benchmarks.jl`
- User has push access to `feature/multi-dataset-benchmark` on `nzy1997/ParametricDFT.jl` (or will use a fork-and-PR workflow for Phase 3)

## Phase 1: Extract and Create New Repo

1. Clone `nzy1997/ParametricDFT.jl` to a temporary directory (e.g., `/tmp/benchmark-extract`).
2. Checkout `feature/multi-dataset-benchmark`.
3. Run `git filter-repo --subdirectory-filter examples/benchmark --force` to rewrite history so that benchmark files are promoted to the repo root and only commits touching those files survive. The `--force` flag is required because the clone has a remote configured.
4. **Verify the extraction:**
   - `ls` confirms expected files at repo root (`config.jl`, `data_loading.jl`, etc.)
   - `git log --oneline` shows only benchmark-related commits
   - No unrelated files leaked through
5. Create `zazabap/ParametricDFT-Benchmarks.jl` as a public repo on GitHub via `gh repo create`.
6. Add the new repo as remote and push the filtered history.

## Phase 2: Update Files in New Repo

### Project.toml

Change the ParametricDFT dependency from a relative path to a git URL. This requires Julia 1.11+:

```toml
[deps]
ParametricDFT = "..."

[sources]
ParametricDFT = {url = "https://github.com/nzy1997/ParametricDFT.jl", rev = "main"}
```

### .gitignore

Replace the current `.gitignore` with:
```
data/
Manifest.toml
```

This removes `results/` from the ignore list (so results are tracked) and adds `Manifest.toml` (environment-specific, regenerated by `Pkg.instantiate()`).

### Manifest.toml

`Manifest.toml` is not tracked in the original repo, so `filter-repo` will not carry it over. Verify it is absent after extraction. The updated `.gitignore` will prevent it from being tracked in the future.

### README.md

- Remove references to being a subdirectory of ParametricDFT.jl.
- Update project paths from `--project=examples/benchmark` to `--project=.`.
- Add a link back to `nzy1997/ParametricDFT.jl`.
- Update `run_all.sh` usage instructions.

### run_all.sh

The script has several hardcoded paths that assume it lives inside `examples/benchmark/` of the parent repo. Specific changes needed:

1. **Usage comments** (lines 6-7): Change `bash examples/benchmark/run_all.sh` to `bash run_all.sh`.
2. **`REPO_DIR` computation** (line 26): Remove the `REPO_DIR="$(cd "$SCRIPT_DIR/../.." && pwd)"` line and the `cd "$REPO_DIR"` line. The repo root IS the script directory, so `cd "$SCRIPT_DIR"` suffices.
3. **Julia invocation** (line 79): Change `julia --project=examples/benchmark "examples/benchmark/$SCRIPT"` to `julia --project=. "$SCRIPT"`.
4. **Results path reference** (line 100): Update to reflect new structure.
5. **Dataset data path references** (lines 17-18): Update `examples/benchmark/data/` to `data/`.

### Run scripts

`run_quickdraw.jl`, `run_div2k.jl`, `run_clic.jl` use relative paths via `config.jl` and should work as-is. Verify during implementation.

### Results

After `filter-repo` completes, `results/` will NOT be present in the new repo because it was gitignored in the original. Manually copy `results/` from the original working tree at `/home/claude-user/ParametricDFT-fresh/examples/benchmark/results/` into the new repo root, then `git add results/` and commit. This includes:

- `results/smoke/`, `results/moderate/`, `results/heavy/` — per-preset metrics, trained models, plots
- `results/plots/` — cross-dataset plots
- `results/cross_dataset_summary.csv`, `results/timing_summary.csv`
- `results/*.log` — run logs

## Phase 3: Submodule in Parent Repo

1. Remove `examples/benchmark/` directory from `ParametricDFT.jl` (on `feature/multi-dataset-benchmark`).
2. Add the new repo as a submodule:
   ```bash
   git submodule add https://github.com/zazabap/ParametricDFT-Benchmarks.jl examples/benchmark
   ```
3. Commit the `.gitmodules` file and submodule pointer.

After this, cloning with benchmarks:
```bash
git submodule update --init examples/benchmark
```

**Rollback:** If the submodule addition causes issues, revert with `git revert <commit>` to restore the previous state.

**Note:** If `zazabap` does not have push access to the `feature/multi-dataset-benchmark` branch on `nzy1997/ParametricDFT.jl`, this phase should be done via a fork-and-PR workflow instead.

## Out of Scope

- No changes to `src/`, `test/`, or core library code.
- No changes to CLAUDE.md safety rules.
- No data files committed anywhere.
- No Julia General registry registration of ParametricDFT.jl.
- CI/CD for the new repo (deferred to a future task).
